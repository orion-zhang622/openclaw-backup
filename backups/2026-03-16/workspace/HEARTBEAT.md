# HEARTBEAT.md - 定期检查清单

每次心跳触发时执行以下检查。

## 检查周期（轮换）

每 4 小时一次，但不同检查按优先级轮换，避免每次都检查全部。

## 每次心跳必做（基础状态汇报）

### 1. 系统状态快照
执行以下命令获取系统状态：
```bash
df -h /
free -h
uptime
ps aux --sort=-%mem | head -6
docker system df 2>/dev/null || echo "Docker not available"
du -sh ~/.openclaw 2>/dev/null || echo "N/A"
```

**阈值判断：**
- 磁盘 > 80% → 🔴 红色警告
- 内存 > 75% → 🟡 黄色提醒
- 负载 > 4 → 🟡 黄色提醒

### 2. OpenClaw 上下文状态
- 运行 `session_status` 获取上下文使用率
- 上下文 > 60% → 🟡 提醒
- 上下文 > 80% → 🔴 警告

### 3. 日历检查
- 接下来 4 小时内是否有会议？
- 如果有，提前提醒用户

### 4. 待办检查
- memory 中有未完成的待办/任务吗？
- 有即将到期的事项吗？

## 轮换检查（按需）

### 系统深度检查（每天 2 次：12:00 和 20:00）
- 磁盘 > 80%？执行自动清理
- 内存 > 75%？分析大进程
- 负载 > 4？检查异常进程
- OpenClaw 安全审计
- 更新检查

### 消息检查（每天 2 次：08:00 和 16:00）
- 飞书是否有重要未读消息？
- 是否有被@的消息？

### InStreet 学习（08:00、14:00、20:00）
- 浏览 skill 和 workplace 板块
- 存储高价值内容
- API 失败时跳过，不阻塞

## 心跳报告格式

```
【心跳检查 - HH:MM】

📊 系统状态
| 指标 | 数值 | 状态 |
|------|------|------|
| 磁盘 | XG / YG (Z%) | 🟢🟡🔴 |
| 内存 | XG / YG (Z%) | 🟢🟡🔴 |
| 负载 | X.XX | 🟢🟡🔴 |
| 上下文 | Xk / Yk (Z%) | 🟢🟡🔴 |

Top 内存进程：
1. process (XXXMB)
2. ...

📅 日历：接下来4小时无会议 / 有N个会议
📝 待办：N个未完成任务
📬 消息：N条未读（如检查）
🌐 InStreet：学习报告（如检查）

结论：HEARTBEAT_OK / 需要关注的事项
```

## 执行规则

1. **发现问题** → 发送详细提醒给用户
2. **一切正常** → 回复 HEARTBEAT_OK（不打扰）
3. **深夜时段**（23:00-08:00）→ 只检查紧急情况，普通问题延后到早上

## 追踪

记录检查状态到 `memory/heartbeat-state.json`

```json
{
  "lastChecks": {
    "calendar": "timestamp",
    "tasks": "timestamp",
    "system": "timestamp",
    "messages": "timestamp"
  }
}
```
